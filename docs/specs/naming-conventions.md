# Naming Conventions

> Adapted from the [Scind specification](https://github.com/scinddev/scind). Xcind uses the same naming patterns with configurable URL template variables.

---

## Application Requirements

For an application to work well within a workspace, it should:

1. **Include a `.xcind.sh`**: This file defines the application's configuration — which compose files to use, environment files, and proxy exports.
2. **Use environment variables for external service URLs**: Don't hardcode hostnames for dependencies. Use the injected `XCIND_*` variables (see [Environment Variables](./environment-variables.md)).
3. **Expose ports without host bindings**: Use `ports: ["8080"]` not `ports: ["8080:8080"]` to avoid conflicts.
4. **Use relative volume paths**: Ensure builds and mounts work regardless of absolute path.

## The Configuration File (`.xcind.sh`)

The `.xcind.sh` file is the interface between the application and the workspace system:

- **Owned by**: Application developers (committed to the application's git repository)
- **Application name**: Inferred from the directory name (no explicit name field required)
- **Purpose**:
  - Declares compose files, env files, and proxy exports
  - Configures hooks and workspace settings
- **Consumed by**: `xcind-compose`, `xcind-config`, and the hook pipeline

## Naming Conventions

- **Workspace names**: Lowercase alphanumeric with hyphens (e.g., `dev`, `feature-x`, `pr-123`)
- **Application names**: Lowercase alphanumeric with hyphens, inferred from directory name
- **Export names**: The key portion of `XCIND_PROXY_EXPORTS` entries

### Hostnames

Generated from configurable URL templates (see [Configuration Reference — URL Template Variables](../reference/configuration.md#url-template-variables)):

| Mode | Template | Example |
|------|----------|---------|
| Workspaceless | `{app}-{export}.{domain}` | `myapp-api.localhost` |
| Workspace | `{workspace}-{app}-{export}.{domain}` | `dev-backend-api.localhost` |
| Workspaceless apex | `{app}.{domain}` | `myapp.localhost` |
| Workspace apex | `{workspace}-{app}.{domain}` | `dev-backend.localhost` |

### Internal Aliases (Workspace Network)

Generated by `xcind-workspace-hook` using `XCIND_WORKSPACE_SERVICE_TEMPLATE`:

| Template | Example |
|----------|---------|
| `{app}-{service}` (default) | `frontend-web`, `backend-api`, `shared-db-db` |

### Traefik Router Names

| Mode | Template | Example |
|------|----------|---------|
| Workspaceless | `{app}-{export}-{protocol}` | `myapp-api-http` |
| Workspace | `{workspace}-{app}-{export}-{protocol}` | `dev-backend-api-http` |
| Workspaceless apex | `{app}-{protocol}` | `myapp-http` |
| Workspace apex | `{workspace}-{app}-{protocol}` | `dev-backend-http` |

### Docker Compose Project Names

Generated by `xcind-naming-hook`:

| Mode | Pattern | Example |
|------|---------|---------|
| Workspaceless | `{app}` | `frontend` |
| Workspace | `{workspace}-{app}` | `dev-frontend` |

### Apex (Primary Export)

The first export in `XCIND_PROXY_EXPORTS` is treated as the primary (apex) export. It receives:

- An **apex hostname**: `{app}.{domain}` or `{workspace}-{app}.{domain}`
- An **apex Traefik router**: Routes to the same port as the primary export

To disable apex URL generation, set the apex URL template to an empty string in `.xcind.sh`.

## Collision Warning

Docker Compose project names, Traefik router names, volume names, and network names are derived from the naming patterns above. Creative naming that produces identical concatenations could cause conflicts:

- **Traefik routers**: Conflicting router names cause routing failures
- **Docker volumes**: Conflicting volume names could cause data to be shared unexpectedly
- **Docker networks**: Conflicting network names could connect unrelated services

Example collision: workspace `dev-front` with app `end` and workspace `dev` with app `frontend` both produce project name `dev-frontend`.

Follow the lowercase-alphanumeric-with-hyphens convention and avoid names that could produce ambiguous concatenations.

## Git Strategy

**Application repositories**:
- `.xcind.sh` is committed to the application repository
- Compose files are committed to the application repository

**Generated files** (gitignored):
- `.xcind/` — generated override files and cache

**Workspace repository** (optional):
- `.xcind.sh` with `XCIND_IS_WORKSPACE=1`
- Application directories (cloned or as submodules)

Example workspace `.gitignore`:

```
.xcind/
```
