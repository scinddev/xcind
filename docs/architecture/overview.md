# Architecture Overview

> Xcind system architecture and component relationships.
>
> Xcind implements concepts from the [Scind specification](https://github.com/scinddev/scind), adapted for a pure Bash implementation.

---

## Overview

A **workspace** is a logical grouping of Docker Compose-based applications that run together on a single host, sharing internal networking for inter-service communication while maintaining isolation from other workspaces. This enables running multiple complete copies of the same application stack simultaneously (e.g., development, code review, and stable/control environments).

## Goals

The architecture achieves the product vision (see [Vision](../product/vision.md)) through these principles:

- **Application independence**: Individual applications remain unaware of the workspace system. No special labels, naming conventions, or workspace-specific configuration required in the application's own compose files.
- **Pure overlay**: All workspace integration is achieved through Docker Compose override files that are generated by hooks and managed externally.
- **Inter-application communication**: Applications within a workspace can communicate via stable, predictable internal hostnames that don't change based on the workspace name.
- **External access**: A shared reverse proxy (Traefik) routes external requests to the appropriate workspace and service based on hostname.
- **Multiple workspaces**: The same set of applications can be instantiated multiple times with different workspace names, running simultaneously without conflict.

## Networks

### Proxy Network

- **Name**: `xcind-proxy`
- **Scope**: Host-level, shared across all workspaces
- **Purpose**: Connects Traefik to services that need external access
- **Created by**: `xcind-proxy init` or auto-created by the proxy hook

### Workspace Internal Network

- **Name**: `{workspace}-internal` (e.g., `dev-internal`)
- **Scope**: Per-workspace
- **Purpose**: Enables inter-application communication within a workspace using stable aliases
- **Created by**: `xcind-workspace-hook` (lazy, idempotent — created if not exists)

### Application Default Networks

- **Name**: Managed by Docker Compose per application
- **Scope**: Per-application
- **Purpose**: Internal communication between services within a single application
- **Created by**: Docker Compose (automatic)

## Proxy Layer

The proxy layer (Traefik) is a prerequisite for proxied services. A single shared Traefik instance serves all workspaces on the host, routing external requests to the appropriate workspace and service based on hostname.

For bootstrap procedures and lifecycle management, see [Proxy Infrastructure Spec](../specs/proxy-infrastructure.md).

## Docker Compose Integration

Xcind generates Docker Compose override files via hooks that extend application compose files without modifying them. This implements the pure overlay design (see [ADR-0003](../decisions/0003-pure-overlay-design.md)).

For each application, the registered hooks can generate:

- **`xcind-naming-hook`**: Sets the Docker Compose project `name:` to prevent container/volume/network collisions (e.g., `dev-frontend` in workspace mode, `frontend` in workspaceless mode)
- **`xcind-app-env-hook`**: Injects `XCIND_APP_ENV_FILES` into services via `env_file:` directives
- **`xcind-proxy-hook`**: Generates Traefik labels, `xcind-proxy` network attachment, and context/export labels based on `XCIND_PROXY_EXPORTS`
- **`xcind-workspace-hook`**: Generates workspace network aliases so services across apps can reach each other via the `{workspace}-internal` network

See [Generated Override Files Spec](../specs/generated-override-files.md) for complete details.

## CLI Architecture

Xcind uses three separate Bash scripts rather than a single binary with subcommands.

```
┌────────────────────┐  ┌────────────────────┐  ┌────────────────────┐
│   xcind-compose    │  │   xcind-config     │  │   xcind-proxy      │
│   (main wrapper)   │  │   (config dump)    │  │   (proxy manager)  │
└────────┬───────────┘  └────────┬───────────┘  └────────┬───────────┘
         │                       │                       │
         └───────────┬───────────┘                       │
                     ▼                                   ▼
            ┌────────────────┐                 ┌──────────────────┐
            │  xcind-lib.bash│                 │xcind-proxy-lib.  │
            │  (shared core) │                 │     bash         │
            └────────────────┘                 └──────────────────┘
                     │
          ┌──────────┼───────────────┐
          ▼          ▼               ▼
    .xcind.sh    Compose files   docker compose
    configs      resolution      execution
```

### Responsibilities

- **xcind-lib.bash**: App root detection, config loading, workspace discovery, file resolution, hook pipeline, SHA caching, template rendering
- **xcind-proxy-lib.bash**: Proxy lifecycle management, YAML template rendering for Traefik labels, export entry parsing, port inference
- **xcind-workspace-lib.bash**: Workspace network alias generation for inter-app communication
- **xcind-naming-lib.bash**: Docker Compose project naming for collision prevention

## Configuration Flow

Xcind sources `.xcind.sh` files (not YAML configuration) and uses hook functions (not a generate engine) to produce compose overrides.

```
┌─────────────────────────────────────────────────────────────────┐
│                    Configuration Flow                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ~/.config/xcind/       workspace/            app/              │
│  ┌──────────────┐    ┌─────────────────┐    ┌──────────────┐   │
│  │ proxy/       │    │   .xcind.sh     │    │  .xcind.sh   │   │
│  │  config.sh   │    │ (workspace-wide │    │ (app-specific│   │
│  │  (global)    │    │  settings)      │    │  settings)   │   │
│  └──────┬───────┘    └────────┬────────┘    └──────┬───────┘   │
│         │                     │                    │           │
│         └──────────┬──────────┘                    │           │
│                    │              ┌────────────────┘           │
│                    ▼              ▼                            │
│              ┌───────────────────────────┐                     │
│              │  Hook Pipeline            │                     │
│              │  (naming, app-env, proxy, │                     │
│              │   workspace hooks)        │                     │
│              └─────────────┬─────────────┘                     │
│                            ▼                                   │
│              ┌───────────────────────────┐                     │
│              │ .xcind/generated/{sha}/   │                     │
│              │  compose.naming.yaml      │                     │
│              │  compose.app-env.yaml     │                     │
│              │  compose.proxy.yaml       │                     │
│              │  compose.workspace.yaml   │                     │
│              └───────────────────────────┘                     │
└─────────────────────────────────────────────────────────────────┘
```

### Caching

Hook output is cached using a SHA-256 hash computed from compose file paths and content, `.xcind.sh` content, workspace `.xcind.sh` content (if present), and global proxy config. On subsequent runs with the same hash, xcind replays the cached output instead of re-running hooks. The cache lives at `$XCIND_APP_ROOT/.xcind/generated/`.
